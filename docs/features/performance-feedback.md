> **Module**: Performance Management  
> **Status**: Production Ready  
> **Version**: 1.0

## Overview

The Performance Feedback System enables continuous performance management through structured feedback cycles, competency-based evaluations, and development planning. It supports manager-led feedback with optional peer and self-assessment capabilities.

### Key Benefits

- **Structured Evaluation**: Standardized competency-based assessments across the organization
- **Development Focus**: Links feedback directly to actionable development plans and training needs
- **Manager-Led Model**: Primary feedback from direct managers with optional 360-degree support
- **Data-Driven Insights**: Aggregated reports and analytics for workforce development
- **Integration**: Connected to appraisal cycles, training management, and employee journey

## Key Features

### 1. Feedback Cycles

**Purpose**: Define periodic performance review windows (quarterly, annual, ad-hoc)

**Capabilities**:
- Create cycles with defined start/end dates and deadlines
- Link to formal appraisal cycles (optional)
- Configure feedback type: manager-only, peer-only, self-assessment, or 360-degree
- Cycle statuses: `draft` → `active` → `closed`
- Track completion rates and participation metrics

**Typical Workflow**:
1. HR creates new cycle (e.g., "Q2 2026 Performance Review")
2. HR launches cycle for specific employees/teams
3. System generates feedback requests to assigned reviewers
4. Managers/peers complete feedback submissions
5. HR closes cycle and generates consolidated reports

### 2. Competency Framework

**Purpose**: Define standardized evaluation criteria across the organization

**Default Competencies** (10 seeded):
- **Core**: Communication, Teamwork, Problem Solving, Accountability, Customer Focus
- **Behavioral**: Adaptability, Time Management, Initiative
- **Leadership**: Leadership
- **Technical**: Technical Skills

**Custom Competencies**:
- Create organization-specific competencies
- Categorize by type (core/leadership/technical/behavioral)
- Activate/deactivate without data loss
- Track usage across feedback cycles

**Rating Scale**: 1-5 (Poor → Below Average → Meets Expectations → Exceeds Expectations → Excellent)

### 3. Feedback Requests & Submissions

**Request Generation**:
- Automatically created when HR launches a cycle
- Assigned to specific reviewers (manager, peers, self)
- Includes due dates and reminder tracking
- Prevents duplicate requests (unique per cycle-employee-reviewer)

**Feedback Form**:
- Dynamic form generated from active competencies
- Rating (1-5) required for each competency
- Optional comments per competency
- Overall strengths and development areas (free text)
- Submission triggers completion timestamp and status update

**Statuses**: `pending` → `completed` (or `declined`, `expired`)

### 4. Performance Reports

**Purpose**: Aggregate all feedback for an employee within a cycle

**Report Components**:
- **Overall Score**: Average of all competency ratings across all reviewers
- **Strengths**: Top-rated competencies (auto-extracted from highest scores)
- **Development Areas**: Bottom-rated competencies (auto-extracted from lowest scores)
- **Reviewer Breakdown**: Self/peer/manager ratings separately tracked
- **Manager Summary**: Dedicated section for direct manager's consolidated feedback

**Report Lifecycle**:
1. Generated by HR after all feedback collected
2. Reviewed and finalized by manager/HR
3. Shared with employee (timestamp recorded)
4. Forms basis for development plan creation

### 5. Development Plans

**Purpose**: Convert feedback insights into actionable improvement roadmaps

**Structure**:
- **Plan Metadata**: Title, employee, linked cycle/report, date range, status
- **Goals**: Multiple SMART goals per plan
- **Resources**: Training needs, mentoring, projects, etc.
- **Timeline**: Target dates per goal
- **Progress Tracking**: Goal statuses (not-started → in-progress → completed)

**Goal Fields**:
- Goal description (what to improve)
- Linked competency (which area addresses)
- Action steps (how employee will achieve it)
- Success criteria (how to measure)
- Target date
- Progress notes (ongoing updates)
- Optional link to training requests

**Plan Statuses**: `draft` → `active` → `completed` / `cancelled`

## Workflow Example

### Quarterly Performance Review (Manager-Led Model)

#### Phase 1: Setup (HR)
1. HR creates cycle: "Q1 2026 Performance Review"
   - Start: 2026-01-01, End: 2026-03-31
   - Deadline: 2026-04-15
   - Type: manager-only
2. HR reviews competency library (uses default 10 competencies)
3. HR identifies participants: 50 employees in Sales + Operations

#### Phase 2: Launch (HR)
1. HR launches cycle via `/cycles/:id/launch` API
2. System auto-assigns direct managers as reviewers
3. 50 feedback requests created with due date 2026-04-15
4. Notification emails sent to managers (via edge function)

#### Phase 3: Feedback Collection (Managers)
1. Manager logs in, sees "5 Pending Reviews" in dashboard
2. Opens feedback form for Employee A
3. Rates 10 competencies (1-5 scale):
   - Communication: 4
   - Teamwork: 5
   - Problem Solving: 3
   - ... (7 more)
4. Adds comments: "Excellent collaboration skills, needs improvement on meeting deadlines"
5. Submits feedback
6. Repeat for remaining 4 employees

#### Phase 4: Report Generation (HR)
1. HR monitors completion rate: 48/50 completed (96%)
2. Sends reminders to 2 pending managers
3. Once 100%, HR generates reports for all employees via API
4. System calculates:
   - Overall scores (3.2, 4.5, 3.8, ...)
   - Top 3 strengths per employee
   - Top 3 development areas per employee
5. HR reviews and finalizes reports

#### Phase 5: Feedback Sharing (HR + Managers)
1. HR shares reports with employees (sets `shared_with_employee_at`)
2. Employees receive notification and view reports
3. Manager schedules 1-on-1 meeting to discuss

#### Phase 6: Development Planning (Manager + Employee)
1. Manager and Employee A meet to review 3.2 score
2. Identify development areas: Time Management, Problem Solving
3. Manager creates development plan:
   - Goal 1: Complete time management training by June 2026
   - Goal 2: Lead 2 cross-functional projects to improve problem-solving (by Sept 2026)
   - Resources: Enroll in "Effective Time Management" course, pair with senior analyst
4. Plan status: active
5. Monthly check-ins to update progress

#### Phase 7: Progress Tracking (Ongoing)
1. Employee updates goal status: "in-progress" on training
2. Manager adds notes: "Showing improvement, completed training module 1/3"
3. Training completion auto-linked when course finishes
4. September review: Goal 1 completed, Goal 2 ongoing

## Configuration

### 1. Competency Library
**Default**: 10 seeded competencies (see Database Schema → competency_master)

**Customization**:
- Add organization-specific competencies via API or frontend
- Categorize for filtering (core/leadership/technical/behavioral)
- Deactivate without deleting historical data

**Best Practice**: Limit to 8-12 competencies per evaluation to avoid reviewer fatigue

### 2. Rating Scale
**Fixed**: 1-5 scale (Poor → Excellent)

**Interpretation**:
- 1: Poor (significant improvement needed)
- 2: Below Average (some areas of concern)
- 3: Meets Expectations (solid performer)
- 4: Exceeds Expectations (strong performer)
- 5: Excellent (exceptional/top performer)

**Calibration**: Recommend manager training to ensure consistent rating standards

### 3. Feedback Types
**Supported**:
- `manager-only`: Single reviewer (direct manager)
- `peer-only`: Peer reviewers only
- `self`: Self-assessment only
- `360`: Self + manager + peers + direct reports

**Current Implementation**: Manager-led model (single manager reviewer per employee)

**Extension**: 360-degree support exists in schema, API supports multiple reviewers, frontend TBD

### 4. Report Thresholds
**Strengths**: Top 3 competencies by avg rating
**Development Areas**: Bottom 3 competencies by avg rating

**Customization**: Adjust thresholds in `performance-feedback.service.ts` (e.g., top 20%, bottom 20%)

## Best Practices

### For HR Administrators

1. **Cycle Planning**:
   - Align cycles with business calendars (quarterly, post-project)
   - Avoid year-end bottlenecks by spreading cycles throughout year
   - Set realistic deadlines (2-3 weeks for managers to complete reviews)

2. **Competency Management**:
   - Review competency relevance annually
   - Keep library focused (8-12 core competencies)
   - Provide clear descriptions and examples for each competency

3. **Calibration**:
   - Conduct manager training on rating standards before cycles
   - Review ratings distribution (avoid inflation or deflation)
   - Hold calibration meetings for managers to discuss edge cases

4. **Communication**:
   - Send pre-launch announcement to employees and managers
   - Provide feedback writing guides (constructive, specific, actionable)
   - Celebrate completion milestones publicly

### For Managers

1. **Preparation**:
   - Review employee's goals, KPIs, and previous feedback before starting
   - Gather specific examples for each competency rating
   - Block time in calendar for thoughtful review (15-20 min per employee)

2. **Feedback Quality**:
   - Be specific: "Delivered 3 presentations to clients with zero prep errors" vs. "Good communicator"
   - Balance strengths and development areas
   - Focus on behaviors, not personality traits
   - Make development areas actionable

3. **Post-Feedback**:
   - Schedule 1-on-1 within 1 week of report sharing
   - Co-create development plan with employee (not top-down)
   - Follow up monthly on development plan progress

4. **Avoid Common Pitfalls**:
   - Recency bias (only remembering recent work)
   - Halo/horn effect (one trait colors all ratings)
   - Central tendency (everyone gets "3 - Meets Expectations")
   - Clone syndrome (rating based on similarity to self)

### For Employees

1. **Self-Reflection**:
   - Keep ongoing notes of accomplishments and challenges
   - Review competency definitions before feedback period
   - Consider self-assessment (if enabled) to prime manager discussion

2. **Receiving Feedback**:
   - Approach with growth mindset (feedback = opportunity, not judgment)
   - Ask clarifying questions if ratings or comments unclear
   - Request specific examples for development areas

3. **Development Planning**:
   - Propose concrete goals and timelines (don't wait for manager)
   - Identify resources needed (training, mentoring, stretch projects)
   - Schedule regular check-ins to track progress

4. **Progress Tracking**:
   - Update goal status and notes in system regularly
   - Celebrate completed goals with manager
   - Adjust plan if circumstances change (don't let it become stale)

## Troubleshooting

### 1. Manager Cannot See Feedback Request

**Symptoms**: Manager logs in, no pending requests visible in dashboard

**Possible Causes**:
- Cycle not launched yet (status = `draft`)
- Request assigned to different user (check `reviewer_id` in database)
- Manager's employee ID mismatch (auth user ID ≠ `employees.id`)

**Solution**:
```sql
-- Check if requests exist for manager
SELECT * FROM performance_feedback_request
WHERE reviewer_id = '<manager_employee_id>'
AND cycle_id = '<cycle_id>';

-- Verify manager's employee ID from auth
SELECT id, emp_code, full_name FROM employees
WHERE email = '<manager_email>';
```

### 2. Report Generation Fails

**Symptoms**: "Failed to generate report" error when HR triggers report creation

**Possible Causes**:
- No feedback submitted yet (no rows in `performance_feedback_response`)
- Request status still `pending` (not `completed`)
- Missing competency data

**Solution**:
```sql
-- Check feedback submission status
SELECT pfr.request_id, pfr.status, pfr.completed_at,
       COUNT(pfrsp.response_id) as response_count
FROM performance_feedback_request pfr
LEFT JOIN performance_feedback_response pfrsp ON pfr.request_id = pfrsp.request_id
WHERE pfr.employee_id = '<employee_id>'
AND pfr.cycle_id = '<cycle_id>'
GROUP BY pfr.request_id;

-- Ensure at least 1 completed request before generating report
-- Report service should handle partial data, but verify logic in service layer
```

### 3. Development Plan Goal Not Updating

**Symptoms**: Employee updates goal status, but UI shows old status

**Possible Causes**:
- Frontend caching issue (stale data)
- API authorization failure (employee lacks permission to update)
- Database update failed silently

**Solution**:
- Check browser network tab for API response (200 OK vs. 403/500)
- Verify employee ID matches `development_plan.employee_id` (employees can only update own plans)
- Check server logs for SQL errors

### 4. Duplicate Requests Created

**Symptoms**: Multiple feedback requests for same employee-reviewer-cycle combination

**Root Cause**: Unique constraint violation or race condition during launch

**Prevention**:
```sql
-- Verify unique constraint exists
SHOW INDEX FROM performance_feedback_request
WHERE Key_name = 'unique_review';

-- If missing, add constraint:
ALTER TABLE performance_feedback_request
ADD UNIQUE KEY unique_review (cycle_id, employee_id, reviewer_id);
```

**Cleanup**:
```sql
-- Identify duplicates
SELECT cycle_id, employee_id, reviewer_id, COUNT(*) as count
FROM performance_feedback_request
GROUP BY cycle_id, employee_id, reviewer_id
HAVING count > 1;

-- Keep oldest request, delete duplicates
-- (Manual verification recommended before deletion)
```

- **Transactions**: Report generation and plan creation use transactions to ensure data consistency

### Validation
- **File**: `/backend/src/modules/performance-feedback/performance-feedback.validation.ts`
- **Library**: Zod schemas for all request bodies
- **Approach**: Parse-safe pattern (return validation errors without throwing)

### Testing
- **Unit Tests**: `backend/tests/performance-feedback.service.test.ts`
- **Integration Tests**: `backend/tests/performance-feedback.integration.test.ts`
- **Coverage**: Core service methods, report generation logic, authorization rules

### Performance Considerations
- **Indexes**: Composite indexes on `(cycle_id, employee_id)` for fast filtering
- **Pagination**: TBD (currently returns all results; implement pagination for large orgs)
- **Caching**: TBD (consider Redis for competency library and cycle metadata)

### Security
- **Authentication**: All endpoints require valid Supabase JWT via `requireAuth` middleware
- **Authorization**: Controller layer checks user role (HR/Manager/Employee) before service calls
- **Data Scoping**: Managers can only access own team's data, employees only own data
- **SQL Injection**: Parameterized queries via `mysql2` (no raw string concatenation)
